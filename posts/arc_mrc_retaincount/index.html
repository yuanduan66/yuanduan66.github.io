ARC 和 MRC 赋值过程中的区别

探讨 Objective-C 中 ARC 与 MRC 内存管理方式的差异

Apr 21, 2025
阅读时长: 2 分钟

简介

上周五技术面试的时候，考官出了个题目：

NSLock* arrayLock = GetArrayLock();//全局锁
NSMutableArray* myArray = GetSharedArray();
id anObject;

[arrayLock lock];
anObject = [myArray objectAtIndex:0];
[arrayLock unlock];

[anObject doSomething];

这是一个典型的多线程问题，但是由于最近小半年一直在用swift写系统插件，导致回答问题时，记忆出现了点混沌。虽然竞争资源的问题回答清楚了，但是自己补充的一个引用计数的问题，回答的时候没太大把握。

我当时的补充回答是：“anObject在引用计数方面感觉有点问题，后续可能会导致线程安全性的问题，但是太久没写OC了，判断依据有点模糊了。”。当然这个回答是一个基于经验型的感觉，但是一直没回想起为什么会有这种感觉。

面试结束后，复盘了一下这个问题，突然意识到这种直觉的出处了。引用计数的安全性问题是源于MRC和ARC的处理不一样，而题目也没有指定使用的是ARC和MRC。

在MRC下，任何需要持有的数据必须手动retain，但是在这里，anObject在赋值后没有调用retain。所以，在unlock后，如果其他线程执行了[myArray removeObjectAtIndex:0]之类的处理，就会导致anObject对象销毁，进一步导致野指针问题。

而在ARC下，anObject在赋值过程中会自动retain。在其他线程中从数组中删除对象不会导致anObject销毁。

define i32 @main(i32 noundef %0, ptr noundef %1) #0 {
  %3 = alloca i32, align 4
  %4 = alloca ptr, align 8
  %5 = alloca ptr, align 8
  %6 = alloca ptr, align 8
  store i32 %0, ptr %3, align 4
  store ptr %1, ptr %4, align 8
  %7 = load ptr, ptr @"OBJC_CLASSLIST_REFERENCES_$_", align 8
  %8 = call ptr @objc_alloc_init(ptr %7)
  store ptr %8, ptr %5, align 8
  %9 = load ptr, ptr %5, align 8
  %10 = call ptr @llvm.objc.retain(ptr %9) #1
  store ptr %10, ptr %6, align 8
  call void @llvm.objc.storeStrong(ptr %6, ptr null) #1
  call void @llvm.objc.storeStrong(ptr %5, ptr null) #1
  ret i32 0
}

从编译得到的LLVM代码可以明显看到明显的objc.storeStrong。
